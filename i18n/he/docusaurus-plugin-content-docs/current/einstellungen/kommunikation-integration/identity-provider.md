בהגדרות > Identity Provider אתם מחברים את Klacks לשירות ספריה חיצוני — LDAP/Active Directory או OAuth2/OpenID Connect — כדי לייבא עובדים אוטומטית ו/או לבצע דרכו את ההתחברות.

## כך זה עובד

**LDAP/Active Directory:** אתם מזינים Host, Port (389/636), SSL, Base DN, Bind DN, סיסמת Bind ומסנן משתמשים (Benutzerfilter). מתג „להשתמש בייבוא לקוחות" מפעיל את הייבוא האוטומטי: שם משפחה, שם פרטי, תואר, חברה וכתובת נלקחים ממאפייני הספריה — ב-OpenLDAP וב-Active Directory לעיתים תחת שמות מאפיינים שונים (למשל `uid` לעומת `sAMAccountName`, `street` לעומת `streetAddress`). מיפויי שדות אלה קבועים ב-Backend ואינם ניתנים לעריכה דרך הממשק.

**OAuth2/OpenID Connect:** Client-ID, Client-Secret, Tenant-ID אופציונלי (Azure), כתובות URL להרשאה (Authorization), לטוקן (Token) ול-UserInfo, וכן Scopes — עבור התחברות באמצעות Google, Microsoft או GitHub.

**בדיקת חיבור וסנכרון:** אצל ספקי LDAP/AD, „בדיקת חיבור" מציגה את מספר המשתמשים שנמצאו כולל שמות משתמש לדוגמה. אם ייבוא הלקוחות מופעל, „סנכרן עכשיו" מפעיל התאמה ידנית; התוצאה (עובד/חדש/עודכן/הושבת) מופיעה מיד לאחר מכן כהודעה בדף.

## טוב לדעת

- דף זה נגיש **רק למנהלות ומנהלי מערכת**, ולא לתפקיד Supervisor.
- סיסמת Bind ו-Client-Secret נשמרות מוצפנות ומוצגות בממשק רק במסווה (`***`); בעת שמירה ללא שינוי, הערך הקיים נשמר כפי שהוא.
- יומן סנכרון קבוע (היסטוריה, היסטוריית שגיאות) **אינו** מוצג כרגע בממשק — רק תוצאת הריצה הידנית האחרונה.
- אנשים שאינם נמצאים עוד בספריה אינם נמחקים בעת הסנכרון, אלא מושבתים באמצעות תאריך סיום החברות שלהם; אם הם מופיעים שוב מאוחר יותר, הם מופעלים מחדש אוטומטית.
- מזהה חיצוני מהספריה (LDAP ObjectGUID, או לחלופין ה-Distinguished Name) מונע כפילויות: אם באותו מזהה נתקלים שוב בסנכרון חוזר, האדם הקיים מתעדכן במקום שנוצרת רשומה חדשה.
